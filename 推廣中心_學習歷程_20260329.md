## 專題動機

我一開始做這個專題，不是只想做一個「會跑畫面」的網頁，而是因為我觀察到：很多人需要網站改版、行銷或資安方面的服務，卻很難在時間有限的情況下，**有效率地挑出真正值得聯繫的對象**。傳統做法往往是買名單、亂槍打鳥，或是只靠感覺排序，結果浪費在無效聯繫上的成本很高。

後來我接觸到名單蒐集、網站健檢、CRM 與 AI 文案這些主題時，表面上看起來每塊都有現成工具，但一旦要把「找客戶 → 判斷需求 → 產出素材 → 追蹤成效」串成同一條路，就會發現難點不在單一功能，而在**資料怎麼進、怎麼洗、怎麼算分、怎麼留下證據**。例如：同一個客戶從不同來源重複進來要怎麼去重？網站分析若遇到 CORS，前端能不能穩定拿到內容？AI 生成的訊息若沒有知識庫與流程，很容易變成空泛文案。

因此我想做的，是比較接近「小型推廣作業系統」的東西：讓使用者能從多種來源蒐集潛在客戶、用可解釋的規則做需求與資安傾向評分、用 AI 輔助產出訊息與報告，最後在 CRM 與 KPI 裡看見自己到底做了什麼。這樣專題的重點就不只是「有 AI」，而是**把 AI 放進可驗證、可展示的工作流**裡。

## 學習心得

這次做推廣中心的過程裡，我慢慢發現，真正麻煩的地方往往不是「畫面做不做得出來」，而是**資料與邊界有没有講清楚**：哪些分數代表什麼、哪些結果只能在 Demo 下展示、哪些功能一定要後端才能穩定運作。若只用 `file://` 開 HTML，很多 fetch 與範例資料會失敗；若沒有後端代理，公開 API 的 CORS 就會讓「網站分析」這類功能在瀏覽器裡卡死。那一刻我才比較具體地理解：**所謂全端思維，很多時候是在處理「誰能發請求、請求怎麼被允許」這種邊界問題。**

另外，模組化對我來說不再是課本上的口號，而是專案變大之後唯一還能改得動的方法。名單去重、合規節流、網站分析、AI、CRM、報告、跟進排程如果全部寫在同一支檔案裡，後面每加一個欄位都像在拆炸彈。把責任拆開之後，我比較敢針對單一模組做測試與調整，也比較清楚「資料從哪裡進、從哪裡出」。

我也更清楚自己偏好的工作方式：除了把功能做出來，我會想把**測試、文件、展示路徑**一併想好——例如 Demo 模式、整合測試腳本、KPI 儀表板，都是為了讓學習歷程或評審在固定步驟下就能重現成果。這很花時間，短期看起來不如一直加新功能有成就感，但長期來說，這才是專題能不能被「相信」的關鍵。

整體來說，這個專題帶給我的不只是技術操作，而是**怎麼說明自己的系統做到哪裡、哪裡還依賴外部金鑰或平台政策、哪裡可以用測試與數據佐證**。這也是我認為在學習歷程裡最值得留下來的一部分。

## 專題總覽

推廣中心不是單一「聊天機器人」或單一「爬蟲腳本」，而是把**名單蒐集、資料清洗、網站與資安訊號分析、AI 文案與分類、CRM 管線、報告輸出、後端 API（代理／寄信／OAuth／追蹤）**放在同一套架構裡。我有明顯感受到，最難的不是把按鈕做完，而是：**評分規則能不能對得上畫面上的說明？沒有 API 金鑰時，展示會不會立刻崩盤？**

因此這份學習歷程刻意對齊「能講清楚邊界」的寫法，重點放在三件事：第一是**系統怎麼拆模組、資料怎麼流**；第二是**三到五個代表流程**（從進名單到出報告）；第三是**我怎麼用測試與腳本，讓「後端有起來」這件事可以被檢查**。

**目前可引用的測試數字（以本機 `npm test` 為例）：**

| 項目 | 數量／說明 |
|------|------------|
| 單元測試（`node --test`） | 11 項，涵蓋 API 參數驗證、靜態路徑安全、去重邏輯 |
| 整合測試（`scripts/run-tests.js`） | 18 項檢查，含 `/api/config`、代理、靜態頁、追蹤與 KPI、關鍵 `js/` 模組檔是否存在 |

## 這份報告想呈現的重點

不是只有功能列表，而是我怎麼把一個「行銷／接案輔助」類專題做成**可操作、可測試、可展示**的系統：評分有對應到實際檢查項、報告與匯出能當提案素材、KPI 與追蹤能回答「做了之後發生什麼事」。

我在這個專題裡學到很重要的一點是：**畫面會動，不代表商業判斷就成立；AI 會寫字，不代表就接得上真實產品與合規節奏。** 所以我把知識庫、節流、黑名單、跟進排程與報告模板放進同一條路，而不是讓 AI 變成孤立的玩具。

## 一、為什麼做推廣中心

我一開始並不是直接想做「大而全的 SaaS」，而是從幾個實際痛點出發：名單從哪來？進來之後怎麼避免髒資料？看到官網之後，能不能用**可解釋的規則**推估「網站優化／資安健檢」需求？最後，能不能把力氣花在**高潛力客戶**上，而不是平均對所有人同一套話術？

做到後面我發現，真正的難點在於**整合**：Google Places 或招募頁只是一個入口，後面還要去重、合規、分析、CRM 狀態、報告輸出與（選用的）實際寄信。若沒有清楚分層，很快就會變成「改一個欄位，整頁一起壞」。

**我給自己訂的三個目標：**

1. **先把資料與名單流程做穩**：去重、驗證、合規節流，讓「進 CRM 的客戶」是有一定品質的。  
2. **讓分析與評分「說得出根據」**：對應 `website_need_score`、`security_need_score`、`lead_score` 等欄位，並在說明頁交代加權與限制（見 `pages/scoring-explained.html`）。  
3. **不要只靠感覺驗收**：用單元測試守住 API 與路徑安全，用整合測試確認後端起來後關鍵模組與頁面可讀，並用 Demo 模式讓展示不依賴正式金鑰。

**圖 1　推廣中心系統結構（文字版）**

```
使用者（app-new.html）
    ↓
js/ 前端模組：蒐集、去重、分析、AI、CRM、報告、跟進…
    ↓（必要時）
server/backend-server.js：/api/proxy、/api/send-email、OAuth、追蹤、KPI…
    ↓
輸出：報告 HTML／CSV／列印、KPI 儀表板、操作日誌
```

## 二、代表案例 1｜名單蒐集與去重（資料進得來、留得下）

這個案例最適合當「入口示範」，因為它能直接驗證：**使用者能不能從不同來源把客戶放進同一個池子，而且不會立刻被重複資料搞亂。**

| 項目 | 說明 |
|------|------|
| **輸入** | 手動新增、Google Maps（需 Places 金鑰）、招募關鍵字、社群／公開資料線索等（依專案目前實作為準）。 |
| **做法** | 前端 `js/` 各蒐集模組產生候選名單 → `js/data-deduplication.js` 等處理 email／公司名正規化與重複合併 → 寫入 CRM（`js/crm-database.js`）與 localStorage。 |
| **結果** | 在「目標客戶」列表可看出去重後的客戶；分數與標籤後續可由分析與 AI 模組更新。 |
| **我學到什麼** | 名單系統的第一個敵人通常不是「演算法不夠帥」，而是**髒資料與重複資料**；先把清洗規則講清楚，後面的評分與 AI 才站得住腳。 |

**建議截圖：** 主畫面 `app-new.html` 的「發現客戶」與名單區（檔名例如 `01-discover-prospects.png`）。

## 三、代表案例 2｜網站分析與評分（把「感覺」變成欄位）

若只有「網站看起來舊」這種主觀描述，很難在學習歷程裡辯護。此專題的做法是：盡量用**可對照的檢查項**（標題與描述、表單、HTTPS、標頭線索等）產出分數與文字摘要，讓使用者能解釋「為什麼這家可能被優先聯繫」。

| 項目 | 說明 |
|------|------|
| **輸入** | 客戶官網 URL、必要時經後端代理抓取 HTML。 |
| **做法** | `js/website-analyzer.js` 等模組解析內容與回應標頭，計算 `website_need_score`、`security_need_score`，並整理 `leadSignals`。 |
| **結果** | 客戶列上出現分數與簡要說明；可與 `pages/scoring-explained.html` 對照。 |
| **我學到什麼** | **CORS 與抓取失敗是常態**，所以要嘛後端代理，要嘛在 UI 上清楚提示限制；評分也要老實交代「沒抓到資料時發生什麼事」。 |

**建議截圖：** 單一客戶分析結果或評分說明頁（`02-website-scores.png`、`07-scoring-explained.png`）。

## 五、代表案例 4｜CRM、報告與 KPI（把行動與成效留下來）

若只有「產生一份 PDF/HTML」但沒有管線狀態與事件追蹤，專題很容易停在展示層。此專題嘗試把 **Pipeline、互動紀錄、跟進排程** 與 **KPI／追蹤像素** 串起來，讓「我有聯絡、有開信、有點擊」這類敘事有地方放。

| 項目 | 說明 |

| **我學到什麼** | **行銷系統最後一定要能回答「然後呢」**——客戶在哪一階段、下一步誰要做什麼、成效事件是否可追溯。 |

## 六、代表案例 5｜後端、安全與測試（讓「能跑」變成「可檢查」）

## 七、驗證、測試與品質門檻

我很在意的一點，是不要把每個功能都寫成「已完美上線」。例如：沒有 SMTP 設定就無法真寄信、沒有 Google API 金鑰就無法真搜尋地點——這些都應該在文件與畫面上**被標成依賴條件**，而不是假裝不存在。

下表用類似「成熟度標籤」的方式，說明本專題目前怎麼看待不同區塊（實際邊界仍以程式與 README 為準）：

| 模組／能力 | 類型 | 信任等級 | 說明 |
|------------|------|----------|------|
| API 參數驗證 | 後端／函式庫 | 高 | 有單元測試直接覆蓋。 |
| 靜態檔 path traversal 防護 | 後端 | 高 | `resolveStaticPath` 有單元測試。 |
| 名單去重與正規化 | 前端 | 中高 | 有單元測試；實際名單品質仍依來源而變。 |
| 網站分析／評分 | 前端＋（常需）代理 | 中 | 受 CORS、目標站結構、取樣頁面影響；分數為啟發式指標。 |
| AI 生成文案 | 前端規則為主 | 中 | 偏展示與輔助；正式商務仍需人工審稿。 |
| 實際寄信／OAuth | 後端＋設定檔 | 視設定 | 需 `backend-config.json` 與平台申請；未設定時應優雅降級。 |
| Demo 模式 | 展示用 | Demo | 範例資料與流程見 `demo/`。 |

**一句話總結：** 測試數字不是拿來炫技，而是證明「核心假設（後端起來、API 有回、檔案路徑安全、關鍵檔在）」沒有在整理專題的過程中壞掉。

## 八、輸出與展示價值

推廣中心不只會在畫面上顯示文字，還把結果往 **HTML 報告、CSV、列印／PDF 流程、KPI 儀表板**送。對學習歷程來說，這代表專題不是只有「按得到按鈕」，而是**拿得出可以給別人看的成品**。

| 輸出類型 | 用途 |
|----------|------|
| HTML 報告 | 健檢、SEO、資安、摘要等，可作提案附件。 |
| CSV／JSON 匯出 | 備份名單、交換資料、與其他工具銜接。 |
| KPI 儀表板 | 把事件與管線狀態視覺化，方便口頭報告時講「我做了什麼」。 |
| 操作日誌 | 留下關鍵操作紀錄，方便說明製作與測試過程。 |

## 九、專題價值、限制與後續方向

我認為推廣中心目前最有價值的地方，不是「已經等於商用 SaaS」，而是它已經有一條**可講述的閉環**：名單 → 分析與評分 → AI 與報告 → CRM 與追蹤 → 用測試與 Demo 支撐展示。

**目前做得比較穩的部分**

- 主流程介面 `app-new.html` 與多個子頁（`pages/`）。  
- 模組拆分於 `js/`，後端集中於 `server/backend-server.js`。  
- 單元測試＋整合測試可一鍵執行（`npm test`）。  
- Demo 模式與範例資料（`demo/`）。

**目前仍要誠實面對的限制**

- 外部 API、寄信與 OAuth 仍依設定與平台政策；未設定時功能會降級。  
- 網站分析對動態渲染過重的站可能不完整。  
- 資料主要在瀏覽器本地，非多人協作雲端架構。

**下一步可延伸**

- 更多來源與匯入格式、行動版介面。  
- 更細的 A/B 測試與開信分析。  
- 若升級為正式產品，需另規畫帳號、權限與伺服器端資料庫。

## 十、學習反思與結語

這個專題讓我最明顯的改變，是我開始分得清楚「把功能做出來」和「把系統做出說服力」的差別。以前可能會先在意畫面炫不炫，現在我會多問：**沒有金鑰時能不能講故事？評分能不能對照文件？別人能不能照 README 重現？**

模組化的意義對我來說也更具體：不只是檔案分開，而是**責任分開**——蒐集、清洗、分析、AI、CRM、報告、後端 API 各自該回答的問題不同，硬塞在一起只會讓後續維護與測試更痛苦。

若用一句話總結推廣中心，我會說：這不是只有把幾個行銷關鍵字和 AI 文案塞進同一個資料夾，而是我試著用**可測試的後端、可解釋的評分、可展示的 Demo**，去做一個知道自己依賴什麼、也願意交代邊界的學習歷程專題。
